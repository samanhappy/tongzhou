# 同舟 · 系统架构（v0.2）

> 技术栈：Next.js（前端 + BFF） + Node 后端 + PostgreSQL
> 部署：阿里云 / 腾讯云（按国内业务合规）
> 状态：草案，待评审
> **v0.2 变更**：基于 04 文档 v0.2（C 路线），删除微信小程序与微信支付两条入站链路；新增第三方付费工具 Webhook 入口。

---

## 1. 架构原则

1. **简单优于复杂**：MVP 阶段单体优先，可拆但不预拆。
2. **后端模块化单体（Modular Monolith）**：按领域（Track / Cohort / Member / WeChat / Billing）划分模块，对内是模块，对外是同一进程。
3. **租户隔离深入到 DB 层**：见 02-数据模型 RLS。
4. **关键操作走异步**：消息推送、视频转码回调、AI 推理走队列。
5. **微信集成层尽量薄**：所有微信交互封装在一个 `wechat-gateway` 模块里，业务代码不直接调微信 API。

## 3. 模块划分（后端 / Modular Monolith）

```
apps/
  web/                   # Next.js (创作者后台 + 平台后台)
  m/                     # Next.js (学员端 H5 + PWA)
  api/                   # Node 后端（Fastify 或 NestJS）
    src/
      modules/
        identity/        # 平台用户、租户、登录、权限
        tenant/          # Tenant 元数据、品牌、订阅
        track/           # Track / Stage / Session
        cohort/          # 班级、报名、排程
        member/          # 学员档案、unionid
        content/         # 视频、资料
        progress/        # 进度、签到
        submission/      # 作业、反馈
        community/       # 群绑定、群运营助手
        wechat/          # ⭐ 微信网关（订阅消息 / OAuth2 / JS-SDK）
        ingress/         # ⭐ 学员入站：CSV 导入 + 第三方付费 Webhook 适配
        billing/         # 同舟自身 SaaS 订阅（toB，向创作者收费）
        notification/    # 通知中心（公众号 / 站内 / 短信）
        ai/              # AI 编排（作业初稿、群日报）
        platform/        # 平台运营专用
      shared/
        db/              # 连接、tenant_id 中间件、RLS
        queue/           # 队列封装
        observability/   # 日志、metrics、trace
        errors/          # 统一错误模型
      bootstrap.ts
  workers/               # 异步任务消费者
  packages/
    types/               # 共享类型（zod schema）
    config/              # 配置加载
```

## 5. 技术选型 ADR 汇总

### ADR-001 单体 vs 微服务

**决策：** MVP 用模块化单体。
**理由：** 团队规模 1-3 人；事务边界清晰；过早拆分会拖慢迭代。
**触发拆分的信号：** 单模块独立部署需求出现 / 某模块明显高负载 / 团队规模 > 8。

### ADR-002 ORM 选型：Prisma vs Drizzle

**决策：** Drizzle（首选）。
**理由：**
- 更接近裸 SQL，便于处理 `SET LOCAL app.tenant_id`
- 类型推导优秀
- 迁移工具足够轻
**代价：** 生态比 Prisma 小，需自己搭一些工具。
**备选：** 若团队对 Prisma 更熟悉，也可，性能开销可接受。

### ADR-003 队列：Redis Stream vs RocketMQ

**决策：** MVP 用 Redis Stream，必要时升级到 RocketMQ。
**理由：** 流量很小，Redis 已经在用，无需额外组件。

### ADR-004 视频云：阿里云 VOD vs 腾讯云 VOD

**决策：** 阿里云 VOD 起步（与 OSS / 短信整套打通），保留 provider 抽象。
**关键能力：** 自适应转码、带账号水印的 playAuth、防盗链。

### ADR-005 鉴权方案（v0.2 调整）

**决策：**
- 创作者后台：邮箱+密码 + Magic Link 备选，session cookie + CSRF。
- 学员 H5：
  - 微信内：公众号 OAuth2 静默授权拿 openid → 关联 member（若已通过手机号预录入则合并）
  - 微信外：短信验证码登录
  - 自家 session token（cookie + localStorage 双轨），14 天滑动过期
- 平台后台：强制 2FA。
**令牌格式：** session ID 入 Redis（不发 JWT），便于即时撤销。

### ADR-006 部署平台

**决策：** 阿里云容器服务 ACK / 函数计算混用：
- API、Web、Worker → ACK
- 微信回调入口 → 也走 ACK，避免冷启动延迟
- 静态资源 → OSS + CDN

### ADR-007 文件存储

**决策：** 业务上传走 OSS 直传（前端拿 STS 临时凭证 → 直接 PUT），后端只接收元数据。

## 6. 安全与合规

### 6.1 网络

- 全站 HTTPS，HSTS
- WAF + 限流（按 tenant_id + IP）
- 公众号回调入口独立子域名，独立白名单 IP（如可控）

### 6.2 数据

- DB 至少三副本（主 + 同城从 + 异地从），每天逻辑备份保留 30 天
- 敏感字段（手机号、密钥）应用层 AES-GCM 加密，密钥走 KMS
- `wechat_apps.credentials` 单独加密
- 密码：argon2id

### 6.3 多租户安全

- DB RLS（见 02-数据模型）
- 接口层 tenant_id 解析与注入由中间件统一负责
- 单元 + 集成 + 模糊测试同时验证"无 tenant 上下文时全部拒绝"

### 6.4 合规

- 备案：必须有 ICP（涉及网站）+ 公安备案
- 内容审核：视频 + 作业 文本/图片 走云厂商内容安全接口
- 个人信息保护法（PIPL）：明文条款 + 数据导出 + 注销

## 7. 可观测性

- 日志：结构化 JSON，每行带 `tenant_id`、`request_id`、`member_id`（如有）
- Metrics：Prometheus 指标，按 tenant 维度的关键 SLO
- Trace：OpenTelemetry，特别覆盖微信回调全链路
- 业务事件流：所有领域事件落 `audit_logs`，便于事后追查
